# 大模型软件测试使用记录

## 使用背景

在本次《软件测试》课程作业中，输出与渲染模块包含较多源码文件、多个输出格式以及一定数量的分支与异常路径。为了提高测试设计效率、补齐高覆盖率所需的白盒测试场景，本次工作中使用了大模型辅助测试分析与文档撰写。

## 使用目标

本次使用大模型的目标主要包括：

1. 快速理解项目结构与模块边界。
2. 辅助梳理输出与渲染模块的功能点。
3. 生成黑盒测试与白盒测试思路。
4. 辅助补充高覆盖率测试用例。
5. 辅助定位 CI 报错和跨平台失败原因。
6. 辅助撰写需求说明、测试计划、测试报告等文档。

## 使用过程记录

### 项目结构理解阶段

使用方式：

- 让大模型先分析仓库结构、模块目录、核心源码文件和现有测试情况。
- 根据分析结果，确认本人负责的模块为 `module3_renderers`。

效果：

- 能较快定位渲染器文件、处理器文件和 HTML 前端资源文件。
- 比手工逐个打开文件更高效。

### 测试计划制定阶段

使用方式：

- 要求大模型同时从黑盒与白盒角度提出测试策略。
- 明确要求黑盒测试包含边界值、等价类、决策表方法；白盒测试包含代码覆盖、基本路径覆盖和数据流分析。

效果：

- 形成了较完整的测试思路框架。
- 便于后续按方法论组织测试用例和实验报告。

### 测试用例补充阶段

使用方式：

- 让大模型根据低覆盖率模块补充测试。
- 重点针对 `console.py`、`html.py`、`processors.py` 中未覆盖分支设计测试。

效果：

- 快速补齐了资源缺失、重采样、多轮重采样、group 隐藏、抽象类异常等场景。
- 覆盖率从不足 95% 提高到 99%。

### CI 问题排查阶段

使用方式：

- 将 GitHub Actions 的报错日志提供给大模型。
- 让其分析是测试问题、类型问题、格式化问题还是功能性回归。

效果：

- 成功识别以下问题：
  - 跨平台路径断言差异。
  - `frame_records` 类型与 `FrameRecordType` 不一致。
  - `str(bytes)` 导致 Windows 路径转义后断言失败。
  - `pre-commit` 未提交格式化结果。

说明：

- 这部分是大模型使用中价值最高的环节之一，因为它能快速结合上下文定位问题。

### 文档撰写阶段

使用方式：

- 使用大模型根据模块源码、测试代码和覆盖率结果生成结构化文档。
- 再结合课程要求进行调整与补充。

效果：

- 显著减少了文档整理时间。
- 文档结构更加完整，表述更加规范。

## 使用中的优点

### 提高效率

- 可以快速总结模块功能点。
- 可以较快生成测试策略草案。
- 对日志和报错的归类分析很高效。

### 辅助补齐遗漏场景

- 人工容易忽略抽象类分支、资源缺失分支、路径差异等细节。
- 大模型在“穷举潜在场景”方面有明显帮助。

### 便于撰写规范化文档

- 可以快速生成较完整的文档结构和内容初稿。
- 对课程作业中“需要详细说明”的部分帮助较大。

## 使用中的不足

### 生成内容需要人工复核

- 大模型生成的测试或文档并不能直接全盘采用。
- 某些断言在跨平台环境中可能不稳定，需要人工修正。

### 可能忽略项目真实约束

- 例如：CI 中的 `pre-commit`、类型检查、平台差异、shell 参数行为等，仍需要结合真实项目环境确认。

### 表述有时过于理想化

- 某些建议在理论上合理，但落到具体仓库时还需按实际代码调整。

## 个人使用体会

本次大模型最有帮助的不是“直接替代测试人员”，而是作为：

- 项目理解助手。
- 测试思路生成助手。
- 日志排查助手。
- 文档整理助手。

尤其是在覆盖率提升和 CI 失败定位时，大模型能够明显减少来回查找源码和日志的时间。但最终测试是否有效、断言是否稳定、文档是否满足课程要求，仍然需要人工负责。

## 结论

在本次输出与渲染模块测试工作中，大模型起到了明显的辅助作用，主要价值体现在：

1. 加快理解模块功能与结构。
2. 帮助建立黑盒/白盒测试体系。
3. 提升测试覆盖率。
4. 帮助排查 CI 与跨平台问题。
5. 提高测试文档撰写效率。

总体来看，大模型适合作为软件测试过程中的辅助工具，而不是替代人工判断的唯一依据。
