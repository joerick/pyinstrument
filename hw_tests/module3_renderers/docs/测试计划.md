# 输出与渲染模块测试计划

## 测试对象

本测试计划对应 `hw_tests/module3_renderers`，测试对象为输出与渲染模块，覆盖：

- `pyinstrument/renderers/base.py`
- `pyinstrument/renderers/console.py`
- `pyinstrument/renderers/html.py`
- `pyinstrument/renderers/jsonrenderer.py`
- `pyinstrument/renderers/speedscope.py`
- `pyinstrument/renderers/pstatsrenderer.py`
- `pyinstrument/renderers/session.py`
- `pyinstrument/renderers/__init__.py`
- `pyinstrument/processors.py`
- `pyinstrument/renderers/html_resources/app.js`
- `pyinstrument/renderers/html_resources/app.css`

## 测试目标

本模块测试的主要目标如下：

1. 验证输出与渲染模块在不同输出形式下功能正确。
2. 验证预处理逻辑是否能正确改变调用树而不破坏语义。
3. 验证关键边界场景，如空会话、大会话、缺失资源、分组隐藏、格式切换等。
4. 从黑盒和白盒两个角度建立较完整的测试体系。
5. 在白盒测试中实现高覆盖率，验证关键分支、基本路径和数据流。

## 黑盒测试策略

黑盒测试不关注源码内部实现细节，重点从输入、输出和用户可见行为角度设计。

### 等价类划分

#### 文本渲染输出

等价类：

- 普通树状输出
- `flat=True` 平铺输出
- `short_mode=True` 简洁输出
- `time="seconds"`
- `time="percent_of_total"`

预期：

- 输出中包含目标描述、关键函数名、时间信息。
- 输出结构符合渲染模式预期。

#### 数据导出输出

等价类：

- JSON 输出
- Speedscope 输出
- pstats 输出
- Session 输出

预期：

- 输出格式合法。
- 关键字段完整。
- 可被进一步消费或反序列化。

### 边界值分析

边界值场景：

- 空 profile
- 只有一个根节点的 profile
- 超过 100000 采样的大 profile
- `resample_interval = 0`
- 时间占比极小节点

预期：

- 空 profile 不崩溃。
- 超大 profile 会触发重采样或跳过重采样逻辑。
- 小节点在过滤策略下表现正确。

### 决策表方法

针对 HTML 渲染和资源加载设计决策表。

条件：

1. `app.js/app.css` 是否存在。
2. 会话是否为空。
3. 采样数是否超过重采样阈值。
4. 是否指定 `resample_interval`。

动作：

- 正常生成 HTML。
- 抛出资源缺失异常。
- 自动重采样。
- 跳过重采样。

针对 `renderers.__init__`：

条件：

1. 用户是否从包入口导入。

动作：

- 公开类可访问。

### 黑盒重点关注点

- 用户是否能看懂文本输出。
- HTML 页面是否能包含完整资源和 session 数据。
- 不同格式输出是否符合消费方预期。
- 包装 API 是否对外暴露正常。

## 白盒测试策略

白盒测试关注源码结构、分支、路径和数据流，目标是高覆盖率与关键逻辑完整验证。

### 代码覆盖策略

覆盖目标：

- 语句覆盖率尽量接近 100%。
- 分支覆盖率尽量覆盖核心判断。
- 重点模块达到高覆盖：
  - `console.py`
  - `html.py`
  - `processors.py`

实施方法：

- 对 `show_all`、`timeline`、`flat`、`color`、`unicode` 等布尔开关分别构造场景。
- 对 `render_frame(None)`、`render()`、`open_in_browser()`、异常分支进行直接测试。
- 对 `processors` 中每个处理器至少覆盖一次正向分支。

### 基本路径覆盖

重点路径：

#### `ConsoleRenderer`

- 正常树状渲染路径。
- `flat=True` 路径。
- `flat=True and timeline=True` 异常路径。
- group 被忽略路径。
- group 被隐藏路径。
- short mode 路径。

#### `HTMLRenderer`

- 正常 HTML 生成路径。
- 资源缺失异常路径。
- 自动重采样路径。
- 多轮重采样路径。
- `resample_interval = 0` 路径。
- `open_in_browser(None)` 路径。
- `open_in_browser(output_filename=...)` 路径。

#### `processors`

- `remove_importlib` 删除路径。
- `remove_tracebackhide` 删除路径。
- `aggregate_repeated_calls` 合并路径。
- `group_library_frames_processor` 中：
  - hide 命中。
  - show 覆盖 hide。
  - 不分组路径。
- `merge_consecutive_self_time` 合并路径。
- `remove_irrelevant_nodes` 过滤路径。
- `remove_first_pyinstrument_frames_processor`：
  - 正常裁剪路径。
  - 初始帧不匹配路径。
  - exec 帧不匹配路径。

### 数据流分析

本模块数据流重点是“会话数据如何经过处理器流入渲染器输出”。

重点分析对象：

#### `Session -> Frame -> Renderer`

- `Session.root_frame()` 输出调用树。
- 调用树经过 `preprocess()`。
- 渲染器将处理后的树转成目标格式。

#### `SpeedscopeRenderer`

- `Frame` 转为 `SpeedscopeFrame`。
- `_frame_to_index` 维护唯一索引。
- `OPEN/CLOSE` 事件成对输出。

#### `PstatsRenderer`

- `Frame -> stats dict`。
- synthetic 节点不进入最终统计。
- 父子调用关系写入 callers 字典。

#### `JSONRenderer`

- group_id 与 class_name 的数据从 `Frame` 属性流入输出 JSON。

### 白盒重点关注点

- 分支逻辑是否都能走到。
- 特殊节点是否按设计处理。
- 同一份会话经过不同渲染器输出时，关键语义是否保持一致。

## 风险与应对

### 跨平台路径差异

问题：

- Windows 和 Linux 下 `shorten_path()` 与路径展示结果不同。

应对：

- 测试断言避免写死某一平台特定路径片段。
- 对 group 描述使用更稳健的匹配策略。

### 大 profile 导致测试慢或脆弱

应对：

- 尽量使用构造的 `Session` 数据而不是实际长时间采样。
- 用 patch 模拟多轮重采样。

### 浏览器打开与文件 IO 副作用

应对：

- 使用 `patch("webbrowser.open")` 隔离副作用。
- 将输出文件写入临时目录。
